# Frontend Code Folder — Explained

This folder contains the **already-built (compiled)** React application — the actual files a browser downloads and runs. There is **no source code** here, only the production output.

When a developer runs `npm run build` on a React project, the output ends up in a `build/` (or similar) folder containing:
- one `index.html`
- bundled JavaScript files (often with hashed names like `main.bcd36112.js`)
- bundled CSS files
- static assets like icons, images, fonts

That output has been copied here as `code/` and is shipped inside the frontend Docker image.

---

## Files

| File | Purpose |
|------|---------|
| `index.html` | The single HTML page the browser loads. It contains an empty `<div id="root"></div>` — the React app mounts itself into that div. |
| `asset-manifest.json` | A map produced by the React build that lists every generated file and its hashed name. Useful for tooling/debugging. |
| `robots.txt` | Tells search-engine crawlers what they may or may not index. |
| `static/` | Folder with bundled JS, CSS, and media (images, fonts) generated by the React build. |

---

## Why ship the pre-built output instead of source?

- The Docker image stays **small** — no Node.js, npm, or `node_modules` are needed in production. Only Nginx and the static files.
- **Faster deploys** — building React code can take minutes; copying static files takes seconds.
- **Simpler container** — Nginx has only one job: serve files.

If you ever change the UI, you would:
1. Edit the React source (in a separate repo / folder),
2. Run `npm run build` there,
3. Copy the new `build/` output into this `code/` folder,
4. Rebuild the frontend Docker image.

---

## Where does the data come from?

The bundled JS calls `/api/...` URLs. Nginx (configured in [`../nginx.conf`](../explanation.md)) forwards those calls to the backend container, which returns JSON.
